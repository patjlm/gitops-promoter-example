# gitops-promoter-example

Multi-component monorepo demonstrating [gitops-promoter](https://github.com/argoproj-labs/gitops-promoter) with the `activePath` monorepo mode from [PR #1337](https://github.com/argoproj-labs/gitops-promoter/pull/1337).

## Repository Layout

- `apps/` — Application components (DRY source, edited by users on `main`)
  - `app-a/` — Helm chart (nginx)
  - `app-b/` — Kustomize (httpbin)
  - `app-c/` — Helm chart (redis config service)
  - `app-d/` — Kustomize (batch CronJob)
- `infra/` — Infrastructure components
  - `infra-e/` — Terraform (generic cloud infra with per-env tfvars)
- `k8s/` — gitops-promoter CRDs applied to the promoter cluster
  - `scm-provider.yaml` — GitHub App credentials
  - `git-repository.yaml` — points to this repo (patjlm/gitops-promoter-example)
  - `promotion-strategies/` — one PromotionStrategy per component with `activePath`
  - `commit-status/` — WebRequestCommitStatus (ci-check + deploy) + TimedCommitStatus (soak timer)
- `setup/` — Local development scripts (minikube + gitops-promoter)
- `gitops-promoter/` — git submodule pointing to PR #1337 branch
- `.github/workflows/`
  - `hydrate.yaml` — custom hydrator for all components
  - `ci-checks.yaml` — validation on PRs and post-merge to active branches
  - `deploy.yaml` — fake deployment that sets a `deploy` commit status on active branches

## Promotion Pipeline

```
development → integration → stage → prod-1 → prod-2 → prod-3
  (auto)        (auto)      (auto)  (manual)  (manual)  (manual)
```

Stage has a 2-minute soak timer before prod promotion.

## Promotion Gates

Each environment requires these `activeCommitStatuses` to pass before promoting to the next:

| Key | Source | What it checks |
|-----|--------|----------------|
| `ci-check` | `WebRequestCommitStatus` | Polls GitHub combined commit status API on the active branch |
| `deploy` | `WebRequestCommitStatus` | Polls GitHub statuses for `context=deploy` set by the fake deploy workflow |
| `timer` | `TimedCommitStatus` | 2-minute soak time (only on `environment/stage`) |

The promoter also injects a synthetic `proposedCommitStatus` on each non-first environment ensuring the previous environment is healthy before promotion.

## End-to-End Flow

1. User pushes a change to `main`
2. `hydrate.yaml` detects changed components, renders per-env output (Helm/Kustomize/Terraform), pushes to `-next` branches with `hydrator.metadata`
3. Promoter opens PRs from `-next` → active branches
4. `ci-checks.yaml` runs on the PR (pre-merge validation)
5. Promoter merges the PR (if `autoMerge: true`)
6. `deploy.yaml` runs on the active branch post-merge, sets `deploy` commit status
7. `WebRequestCommitStatus` polls GitHub API, picks up `ci-check` and `deploy` as success
8. Promoter chains to the next environment (repeats from step 3)
9. At `stage`: `TimedCommitStatus` enforces a 2-minute soak before prod
10. At `prod-*`: PR is opened but `autoMerge: false` — requires human approval

## Branch Conventions

- `main` — DRY source branch (users edit here)
- `environment/<env>` — active branches (shared across components)
- `environment/<env>/<activePath>-next` — proposed branches (per component)

## Key Design Decisions

- No ArgoCD dependency — hydration via GitHub Actions, gating via WebRequestCommitStatus polling GitHub's commit status API
- `activePath` mode — all components share environment branches, each promotes independently
- One WebRequestCommitStatus per gate per PromotionStrategy (each references a `promotionStrategyRef`)
- The `timer` key is auto-generated by TimedCommitStatus (not configurable)
- ArgoCD CRDs must be installed on the promoter cluster even without ArgoCD (the controller watches Application resources)
